## What this is

Windows 11 tray app (.NET 8 WPF) that watches 17 gaming-related Windows settings and either prompts the user on drift or silently auto-applies. Pure user-mode P/Invoke — no drivers, no kernel hooks, no admin needed for read-only monitoring.

## Conventions

- **`DesiredOn = true` always means "gaming-optimized"**, even when that maps to a registry value of 0 (e.g., USB Selective Suspend, Mouse Precision off). UI labels: **Enabled / Disabled** for binary toggles where on/off is intuitive, **Gaming / Default** for value-based ones (System Responsiveness, Network Throttling, USB SS, Games Task Profile).
- Each `Monitor` file is self-contained: read raw, compute desired, yield `DriftItem` with `RawBefore` + `RawDesired` populated. Apply lambda uses `ElevatedRegistry.SetHklmDword` for HKLM, direct registry/API for HKCU.
- `IsMonitored` on `DriftItem` is a flag, **not** a filter. `MonitorService` filters polls to `IsMonitored`; `SettingsWindow.Apply` does not (so user-driven Apply works regardless of Monitor checkbox).
- **Settings UI auto-saves on every preference toggle** + logs to `changes.log` as `[ui]` source. `Window.Closing` also persists. `Cancel` button sets `_suppressSaveOnClose=true` to discard.
- Apply path: `PersistFormToConfig → _store.Save → CheckDrift across all monitors → ChangeApplier.ApplyAndVerifyAsync → ChangeLogger.LogApplyResults("manual"|"auto") → ApplyResultsWindow.Show`.
- Reboot semantics: `RequiresReboot=true` on `HagsMonitor`, `MemoryIntegrityMonitor`, `UsbSelectiveSuspendMonitor`, `SystemResponsivenessMonitor`. Manual Apply triggers a `MessageBox` reboot prompt; auto-apply triggers a non-modal `RebootPendingWindow`.

## Known gotchas

- **Edit tool can fail silently** — `String to replace not found` is reported but the build still passes (old code is valid). After every Edit on a critical block, **Grep to confirm the new code is in place**. This bit me on v0.1.28 (power plan combo).
- **Hardcoded GUIDs are dangerous** — Power Saver's GUID I had as `a1841308-1541-...` was wrong; the user's actual one is `a1841308-3541-...`. The fix was to populate the combo from `Powrprof.EnumerateSchemes()` (`PowerPlanMonitor.ListAvailablePlans()`) instead of relying on hardcoded enums. The legacy `PowerPlanChoice` enum + `ToGuid()` is kept only for back-compat with old configs; new flow uses `DesiredGuid` (string) + `DesiredName` (display cache).
- **`Run.Text="{Binding Foo}"` defaults to TwoWay** — must add `Mode=OneWay` when the source property is read-only. Cost me v0.1.18's broken Apply Results window.
- **HKLM writes always need UAC** — use `ElevatedRegistry.SetHklmDword` / `SetHklmString` / `SetHklmMulti`. The verb is `runas`, which spawns `reg.exe` and gets a UAC prompt. UAC cancel = silent failure → verify catches it via re-read.
- **WPF-UI's FluentWindow combo doesn't expose UIA Expand pattern** — useless for "screenshot the dropdown open" type tests. Verify via the underlying data instead.
- **`Mode=OneWay` matters for Run.Text** (see above) — also for any read-only computed property bound to a Run/TextBlock.
- **Single-instance mutex name is `GamerGuardian.SingleInstance`**. Second instance silently exits via `Shutdown()`. Means `--show-settings` from a second launch *doesn't* re-show the existing window (could be added).
- **Debug build vs installed build state collision** — both use the same `%APPDATA%`, `%TEMP%`, `HKCU\Run` paths. Last writer wins. Open issue at end of last session: should add `#if !DEBUG` around `StartupRegistration.Sync` and the auto-update check so Debug doesn't clobber installed.
